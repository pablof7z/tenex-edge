# Tenex-Edge Messaging

## Agent Messaging

In TENEX, all agent-to-agent communication, user-to-agent messages, and project artifacts flow as cryptographically signed Nostr events; each project has its own nsec for identity; relays are the message bus with no central servers. The term for inter-agent messages is 'mention', not 'direct message'. Mentions are sent via `tenex-edge inbox send --to <agent@hostname> <message>` using positional arguments. When `--message` is omitted or set to `-`, the command reads from stdin, failing fast if stdin is a terminal; one trailing newline is stripped from stdin input. The fully qualified recipient format is `agentSlug@projectSlug`, publishing as a kind:1 event with a NIP-29 `h` tag for the project group and a p-tag for the destination agent's pubkey. Local delivery routes messages by `(pubkey, project)`, so same-agent sessions in other projects do not receive the mention. The agent wire tag was completely removed from the codec: no encoder writes it (Presence, Status, Activity, Mention, Proposal), nothing reads it, and the `agent_slug()` function was deleted. On decode, kind:1 events are classified by structural markers rather than by agent tag: the presence of a p-tag indicates a Mention, and e-tag root/reply markers indicate a TurnReply; other kind:1 events are classified as ambient Activity. The sender slug is resolved from the signer's kind:0 profile via the profile store at routing time (no agent tag on the wire, no owner special case). Removing the agent tag is a breaking wire change: a non-upgraded daemon that reads the (now-absent) agent tag to classify mentions will see them as ambient Activity and not deliver them to peers. The Mention vs Activity classification is decided purely by the presence of the p tag on the kind:1 event. A directed mention routes only if the signer pubkey is a member of the hosted agents, owners, or project group members — otherwise it is dropped. `handle_incoming` must gate Mention routing by skipping events authored by operator/owner keys (not by checking for agent tags), so user-prompt relay echoes never enter the inbox. `fetch_mentions_into_inbox` explicitly skips events with no agent tag to prevent user prompts from entering inbox routing, even if a relay reorders or adds tags that cause a false Mention decode. Notes and approvals to agents are plain natural-language content in a kind:1 event — no structured machine-tags like 'approve' or 'proposal' are needed, because capable agents read natural language and act. Agent-to-agent notes and 'do it' signals are sent via the existing `inbox send` command (a plain kind:1 to the generating session), requiring no new instruction protocol. Injected mentions must carry the sender's session ID (the return envelope) so the recipient knows which session to reply to, since sibling sessions share an author pubkey; the reply handle resolves to the sender's session id if that session is resolvable on the recipient's side, otherwise falls back to slug@project. The from_session field is threaded end-to-end: the Mention domain type carries `from_session: Option<String>`, encoded on the wire as a `from-session` tag and stored as `from_session` in the inbox, so the reply address survives end-to-end. When inbox is eventually replaced by messages + message_recipients in a future migration, the canonical messages table must retain `author_session` (the return envelope) alongside `author_pubkey`, and SendIntent must carry `from_session` so it becomes the stored `author_session`; inbox.from_session must be copied to messages.author_session so the return envelope is never dropped. The dual-write to canonical messages/message_recipients is best-effort (`.ok()`, never failing the legacy path) — the legacy inbox remains the authoritative reader. `add_message_recipient` must be idempotent for the NULL `target_session` case; SQLite treats NULL as distinct in the primary key, so without an explicit upsert guard, relay echo or catch-up refetch would produce unbounded duplicate rows. Mentions can target a particular session when the same agent is running in multiple sessions, using `--recipient <session-id>`; `--recipient` also accepts an agent name when no session is running. The send-message command resolves recipients by matching hash-based session codes, with a fallback order of UUID prefix → hash code → agent slug. Send-message resolves the recipient by session-id prefix across both peer sessions and own sessions, and session-aware sender resolution honors `$TENEX_EDGE_AGENT` rather than falling back to a cross-agent latest-alive-in-project session. When mentioning an agent's kind:0, the nostr EventBuilder must use `.allow_self_tagging()` so that p-tags equal to the author's pubkey are preserved (required for same-agent cross-session messaging). Same-machine delivery does NOT rely on the relay echoing a self-published event back to the publishing connection (relays generally do not): when the recipient pubkey is one this daemon hosts, `rpc_send_message` routes the just-published mention into the recipient session's inbox locally, keyed by the same event id it published; `route_mention_into`/`enqueue_mention` is idempotent on the inbox primary key `(mention_event_id, target_session)`, so a later relay echo (if any) does not double-deliver. Mentions are deduplicated per-agent for AGENT-WIDE (untargeted) mentions only: once an agent has seen an untargeted mention in any session, it never resurfaces in a later session. SESSION-TARGETED mentions bypass the per-agent `seen_mentions` dedup entirely (the per-agent `is_mention_seen` check is gated on `target_session.is_none()`), relying instead on the per-(pubkey, session) idempotency of the inbox primary key, so a session-targeted reply between same-agent sibling sessions is never dropped even if a sibling already marked the event seen. The substrate provides a streaming mention subscription via `tenex-edge subscribe --json`, a long-lived command that emits NDJSON one mention per line; `wait-for-mention` remains as the portable floor for environments that cannot maintain a persistent subscription. A `tenex-edge tail -f` command streams all messages colorized, optionally filtered by project slug; in presence events, the hostname appears in the `@` slot and the project name is shown dimmed in parentheses. The UserPromptSubmit hook creates a new kind:1 Nostr note that is an OP (root event with no e tag), signed by the nsec stored in `~/.tenex/config.json` under the `userNsec` key. It publishes the kind:1 OP to the NIP-29 group by adding an h tag with the project slug, and p-tags the agent pubkey that will process the message. The hook fails open — if userNsec is absent, nsec is invalid, session is not found, or relay publish fails, the hook logs via eprintln and returns Ok(()) rather than propagating the error. Internally, the daemon dispatch routes `user_prompt` to a new `rpc_user_prompt` handler, which resolves the session to obtain `agent_pubkey` and project, parses `userNsec` from config, builds a kind:1 EventBuilder with h (project) and p (agent_pubkey) tags and no e tag, then signs and publishes via `transport.publish_signed`. In `cli.rs`, the user-prompt-submit hook arm extracts the prompt field from stdin JSON, clones `sid` for both `turn_start` and the new RPC, then calls `user_prompt` failing open. An owner-signed note published straight to the production relay (nip29.f7z.io) with no agent tag reaches the live agent's session inbox, with the sender name resolved from the owner's kind:0 profile. Inbound admission of owner-signed notes uses the ownership gate: a p-tagged kind:1 with `session-id` and no `agent` tag is admitted because the author is the owner, routing it to the specified session inbox. (Previously, a kind:1 event with a p tag but no agent tag was decoded as Activity, meaning a user OP would never land in any agent's inbox.) The UserPromptSubmit hook injects the available agents list (who) with what each is doing, so agents automatically know who's reachable without running a command. The `who` command displays agents across all projects by default, regardless of which project directory it is run from; passing `--project <slug>` filters the view to a specific project. When agents span multiple projects, the `who` output groups other-project agents by project name and metadata rather than listing each agent individually. The agent display format uses `agentSlug@hostname`, where the hostname is sourced from the `backendName` property in the `.tenex/config.json` file and must be sluggified to prevent ambiguity when agents address messages to one another; the raw hostname value is preserved unmodified in the Nostr data model and Nostr tags, and slugification is applied only at display time. In the `who` command's compact view, the agent's project is displayed in the secondary dim field previously occupied by the host. In the `who` command's live table view, `slug@host` is shown in the AGENT column and the separate HOST column is removed. (Previously: the agent display format used `agentSlug@hostname` where the hostname was sourced from `backendName`, but `agentSlug@project` was used in the recipient format.) Agents can run `tenex-edge who`, `inbox`, or `inbox send` without specifying `--session`; the session is resolved via the `$TENEX_EDGE_SESSION` environment variable (exported by launchers/plugins) or the current working directory's project. Claude Code and other harnesses have access to an inbox send skill to send messages to other agents. The send-message skill injects `CLAUDE_SESSION_ID` as a documented skill substitution because that id is not available in the Bash environment. The inbox command self-fetches stored mentions from the relay so receive works even in one-shot runs where the engine hasn't had time to accumulate them. Agents reply to inbox messages using the command format: `tenex-edge inbox reply --id <messageId> "<reply text>"`. The daemon looks up the inbox row by event-id prefix when resolving an `inbox reply --id` command. A wait-for-mention command polls the SQLite inbox for incoming mentions, prints them upon receipt, and then exits. Its output includes a reminder to re-run `tenex-edge wait-for-mention` with `run_in_background=true` to receive the next mention. The agent runs `wait-for-mention` via its shell with `run_in_background=true` rather than using shell `&`, so the harness tracks the process and wakes the agent on completion. An idle agent is woken when a background process it launched completes, enabling the `wait-for-mention` flow to work for idle agents. The `wait-for-mention` command performs the same relay self-fetch as `inbox` on startup to handle the engine warmup race, polls the SQLite inbox every 500ms, and has a default timeout of 5 minutes to prevent forgotten background processes from lingering. The `wait-for-mention` instruction is injected via the `UserPromptSubmit` hook, firing exactly once per session (tracked via a temp flag file keyed on `sid`), so the agent can act on it during its first active turn. The session-start hook is not a suitable place for this instruction because no LLM call occurs at session start; the agent is idle and cannot execute commands until a user prompt triggers a turn. The relay `relay.tenex.chat` requires NIP-42 AUTH for reads; subscriptions opened before auth completes get silently closed. Transport::connect forces an auth warm-up fetch before any subscribe. The session routing bug fix must be sequenced after the cwd/who implementation to avoid colliding on the same routing/who code. The `threads --project` command must print the full thread id (not a truncated short_id) so the id can be passed back to `--thread`. `rpc_thread_meta` must return an empty object for a missing thread rather than bare JSON null, because the client's `Option` result interprets bare null as neither ok nor error. Real claude-code, codex, and opencode agents can hold a 3-agent threaded conversation through the refactored daemon (inbox send, --thread reply, threads read) with messages grouped by shared NIP-10 root event in the canonical read model. The Claude Code adapter's `.claude/settings.json` template must include `tenex-edge inbox send` in its Bash allowlist so the agent can reply unattended. The messaging command surface is unified under `inbox`; `send-message` is the same command surface as `inbox send --to`.

## Milestone Scope

Inbound injection of peer messages is included in M1 scope. Message injection is immediate (not deferred).

The singleton daemon actively owns the NIP-29 group for each project it runs sessions in, rather than relying on the relay's implicitly-open groups. On `session_start`, before the engine begins publishing, the daemon ensures the project's group exists and that the starting agent is a member: if the project's group is not yet owned it publishes a kind:9007 create-group followed by a kind:9002 edit-metadata that locks the group `closed` (only members may write) and `public` (anyone may read), then for each new agent it publishes a kind:9000 put-user adding that agent's pubkey as a member — so an agent always finds itself already a member when it publishes presence. All three management events are signed by the operator's `userNsec` (the group creator/admin), validated against `nip29.f7z.io` by `tests/nip29_probe.rs`. The group is locked `public` rather than `private` deliberately: the daemon's single relay connection authenticates as the non-member `tenex-edge-daemon` key, and a `private` group would stop delivering its own agents' events to that connection, whereas `public` keeps reads open while still enforcing membership on writes. Ownership and membership are cached in the SQLite `owned_groups` and `group_members` tables (the daemon is the sole writer), so create/add are idempotent and skipped after the first time, and the cache survives daemon restarts; `reconcile_sessions` re-ensures ownership/membership for revived sessions. The daemon keeps a persistent subscription to the relay-authored group state (kinds 39000 metadata, 39001 admins, 39002 members) scoped by the `#d` tag (group id == project slug) for every project it owns, and `handle_incoming` applies each kind:39002 snapshot as the authoritative membership set (`replace_group_members`). Management is best-effort and fails open: if `userNsec` is unset or invalid, or a relay publish fails, the session still starts (the project's group simply remains unmanaged) — unlike `rpc_user_prompt`, which bails when `userNsec` is missing. Critically, the `owned_groups`/`group_members` cache is updated only when the relay actually accepts the event (an "already exists" rejection on create counts as success); a transient rejection (rate-limited, network) leaves the cache untouched so the next `session_start` retries, rather than poisoning the cache by recording ownership/membership of a group that was never created (which would block the agent's presence writes forever). This requires distinguishing a real publish from a silently-rejected one, since `send_event` resolves `Ok` even when every relay rejects; `transport.publish_signed_checked` surfaces the relay's rejection reason as an error for exactly this gating. Consequence of closed/managed groups: cross-operator peer agents that previously posted into a shared-slug project group merely by using its `h` tag are rejected (`blocked: unknown member`) once the group is formally owned; because a project slug maps to a single relay group id, the first operator to formalize a slug becomes its gatekeeper. This is an accepted tradeoff.
