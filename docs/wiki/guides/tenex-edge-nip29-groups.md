# Tenex-Edge NIP-29 Groups

## NIP-29 Group Ownership and Lifecycle

The singleton daemon maintains an open subscription to NIP-29 groups it owns at all times, scoped by `#d` for owned project slugs, covering relay-authored group state events (kinds 39000, 39001, 39002), riding the existing long-lived subscription. When starting a session in a project that does not yet have a NIP-29 group, the daemon automatically creates the group and locks it to closed+public mode before spawning the session. Creating a NIP-29 group (kind 9007) does not automatically enforce membership; a separate 9002 edit-metadata event with closed+public tags is required to lock the group. The `ensure_group_and_membership` step runs before `spawn_session` in `rpc_session_start`, creating the group if unowned and adding the agent if not a member. `reconcile_sessions` re-ensures group ownership and membership for revived sessions across daemon restarts, restoring the persistent subscription. `handle_incoming` caches kind 39002 member snapshots authoritatively into the `group_members` store. When the local membership cache is empty on a fresh daemon, the group membership check must query the relay's kind:39002 rather than relying solely on the local cache, to avoid false warnings. Group ownership and membership state are persisted in new `owned_groups` and `group_members` database tables via the single-writer `with_store` path. Cache writes to `owned_groups` and `group_members` are gated on confirmed relay acceptance via `publish_signed_checked`, which errors on relay rejection and treats 'already exists' as success; a transient failure (e.g., rate-limit, network) leaves the cache untouched so the next `session_start` retries, preventing cache poisoning. Group creation uses the client-chosen project slug as the `d` tag / group id, which the relay honors as the canonical group identifier; the NIP-29 probe validates that the relay honors this client-chosen group id. The daemon authorizes group management writes by the event's author (the admin/userNsec key) rather than the connection's AUTH identity (the non-member `tenex-edge-daemon` key), as verified empirically on relay29. The live topology verification (Blocker 1: whether relay29 authorizes writes by event-author vs connection-AUTH-identity) is required before the design is fully verified, as nak serve does not enforce NIP-29. Owned NIP-29 groups use `closed`+`public` access control rather than `closed`+`private`, because `private` would blind the daemon's non-member connection to its own agents' reads, while `closed`+`public` enforces membership on writes while keeping reads open. Authorization for routing is determined by the signer's pubkey and its NIP-29 group membership, never by a self-asserted tag; the forgeable agent wire tag was completely removed (not written, not read). A note routes only if the signer is a hosted agent, an owner, or a known member of the project group. Cross-operator peer agents from other operators are intentionally locked out of slugs owned by the first operator as an accepted tradeoff of the closed/managed model. The local ACL system (tenex-edge acl command, pending_agents table, whitelisted-agents.txt, blocked-agents.txt) is removed from the product; admission and membership control is handled entirely by the NIP-29 relay. If a kind:1 codec is ever introduced in the future, local ACL allow/block files and logic will be owned and managed exclusively within that kind:1 codec, not globally. Profile subscription author filters use local hosted agents, owner keys, and group state rather than a local allowlist; the owner-p-tag profile discovery filter is removed from subscriptions because it only existed to populate pending ACL rows. Profile materialization persists delivered profiles directly without branching on local allow/block files. When starting a session with a new agent, the daemon auto-adds the agent as a member to the project's NIP-29 group before the engine starts, so the agent finds itself already added when publishing presence status; this uses `group_put_user` (kind:9000) with no join-request flow. The CLI command to add a member to a project group is `tenex-edge project add <project> <pubkey-or-npub-or-nip05>`, accepting hex pubkeys, npub/bech32, or NIP-05 (user@domain.com) as the pubkey argument. Group creation, locking, and membership events are signed with the `userNsec` key from `.tenex/config.json`. If `userNsec` is unset, the group creation and membership step logs the issue and continues (fail-open), allowing the session to still start. When an agent is not a member of the NIP-29 group on the first UserPromptSubmit of a session, the hook output must include a mandatory ACTION REQUIRED block that forces the agent to surface the warning to the user as a blocking obligation before proceeding. If the daemon connects to nip29.f7z.io for presence events, the NIP-29 relay will require a tenex-edge group to exist (via project edit) before accepting kind:30315 events, otherwise they will be rejected. The `nip29_probe` test is resilient to relay rate-limiting: it retries group creation with backoff and skips gracefully (rather than failing) when persistently rate-limited. The production daemon cutover was verified: an owner-signed note published directly to nip29.f7z.io with no identity tag was delivered to a live agent's session inbox, with the sender name resolved from the owner's kind:0 profile. NIP-29 group management operations (group_create, group_lock_closed, group_put_user) and 39000/39001/39002 group-state subscriptions are relay-level operations outside the DomainEvent flow, existing in the kind1 module but not as fabric domain events; they are a property of the NIP-29 transport/ACL strategy, not the kind1 event codec, and should be refactored to a NIP-29 transport/ACL strategy module.

## Admin Membership and Inheritance

Every NIP-29 group must have mgmt_pubkey and all whitelisted_pubkeys as admins unconditionally. Child groups inherit all admins from their immediate parent group. <!-- [^019f0-2a222] -->
