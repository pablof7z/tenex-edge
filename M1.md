# tenex-edge — MVP 1 (M1) Plan

> Status: draft for review, 2026-06-08. High-level build plan. Single-operator, your keys only.
> No trust model, no coordination/locking. This is Rung 0→1 compressed: durable agent identity +
> awareness + messaging, published onto the fabric (Nostr) from the start, scoped to your own
> whitelisted pubkeys. Built in **Rust on NMP** (`~/Work/nostr-multi-platform`).

---

## 1. What M1 delivers

A single binary, `tenex-edge`, plus Claude Code hooks and a skill, that gives every agent session:

- **Identity** — a per-agent Nostr keypair + a published `kind:0` profile carrying the agent's
  slug and a `host` tag, so any peer can resolve `pubkey → slug` and see which machine it's on. An
  agent is **machine-bound**: the same tool on another machine is a different agent (different key).
- **Presence** — agents heartbeat that they're alive, on which project, in which session.
- **Awareness** — agents' activity is **auto-distilled** from their tool use and published, scoped
  to a project (agents never have to remember to narrate themselves).
- **Status** — a live, replaceable "currently doing X" per project (idle = empty).
- **Mentions** — message another agent, targeting either the agent or a **specific session**
  (because the same agent may be running in several sessions at once).
- **Injection** — mentions addressed to a session are injected into that session's context.
- **Observability** — `tenex-edge tail -f` streams all of it, colorized.

**tenex-edge knows nothing about any host.** It's a standalone substrate with a host-neutral CLI
surface. Claude Code talks to it through hooks and a skill from the outside; the dependency arrow
points one way only (`hooks/skill → tenex-edge`). There is no `pc` and no `claude` concept inside it.

### In scope
Identity/keystore + `kind:0` publishing, project-slug resolution, config, presence heartbeat,
auto-distilled activity, NIP-38 status, mentions with agent- **and** session-level targeting,
inbound injection, peer discovery (`who`), `tail`, the send-message skill, session lifecycle +
liveness reaper.

### Out of scope (deferred)
Coordination/locking (gated on the Q1 collision experiment), cross-person trust/authorization,
non-Claude-Code adapters (Codex/MCP/mobile — the boundary is host-neutral so they slot in later),
**auto-launching a Claude Code when a no-session agent is mentioned** (future; see §8), relay
redundancy, reputation, true cross-device key unification (a Rung-1 concern; see §4).

## 2. Architecture — components and dependency direction

```
        ┌──────────────────────────────────────────────────────────────┐
        │                          tenex-edge                            │
        │                                                                │
        │   cli ── runtime ── domain ── codec(set) ── transport (NMP)    │
        │            │                                      │            │
        │     app state (SQLite):                    NMP kernel:         │
        │     sessions, inbox/queue,                 relays, signing,    │
        │     CC PID, routing                        NIP-42 AUTH, store  │
        │                                            (kind:0 / status    │
        │                                             resolution), LMDB  │
        └───────────────────────────▲───────────────────────────────────┘
                                     │  host-neutral CLI surface
        ┌────────────────────────────┴──────────────────────────────────┐
        │   Claude Code integration (lives OUTSIDE, depends inward)       │
        │   SessionStart · SessionEnd · PostToolUse · UserPromptSubmit    │
        │   hooks + the send-message skill                                │
        └─────────────────────────────────────────────────────────────────┘
```

**Modules (seams, not files):**

| Module | Owns | Knows about |
|---|---|---|
| `domain` | Pure models + events: `Identity`/`Profile`, `Project`, `Session`, `Presence`, `Activity`, `Status`, `Mention`. No Nostr, no kinds, no tags, no host wire-format. | nothing below it |
| `codec` | Maps **every** domain event ⇄ wire envelope, **and** the subscription filters that fetch them. The "kind:1 shape" lives here. | `domain` |
| `transport` | Thin adapter over **NMP** (`nmp-core` + `nmp-network` + `nmp-nip42` + `nmp-store`): publish, subscribe, sign, AUTH, replaceable resolution. | wire events only |
| `runtime` | Session lifecycle, the per-session NMP kernel, liveness/reaper, app state (SQLite), routing/injection. | `domain`, `codec`, `transport` |
| `cli` | Subcommand parsing; thin wrappers over `runtime`. | everything |
| `config` / `identity` / `project` | Read `~/.tenex/config.json`; keystore (slug→keypair); resolve project slug from cwd. | local fs only |

The hard rule that keeps the larger vision open: **`domain` never names a kind or a tag.** It emits
`Mention { to, project, body, target_session? }`; the codec decides that's a `kind:1` with `t`/`p`/
session tags today, or an `nmp-nip29` group message / `nmp-marmot` (MLS) payload tomorrow.

**Two stores, on purpose:** NMP brings its own event store (LMDB) and resolves replaceable events
(`kind:0`, `kind:30315`) so we never hold a stale profile or status. tenex-edge's **SQLite** holds
only *app* state NMP shouldn't know about: session records, the captured CC PID, the per-session
injection inbox, and queued mentions for not-yet-running agents.

## 3. The codec seam

A **codec set** is the active "shape." For *each* domain event type it owns encode (domain → wire),
decode (wire → domain, or ignore), and the subscription filters. M1 ships one set — `kind1` — which
uses standard Nostr event kinds with NIP-29 `h` group tags for project scoping. A future Marmot/MLS
shape remains an additive plug-in selectable by config.

| Domain event | `kind1` codec → wire | Storage semantics |
|---|---|---|
| `Profile` (identity) | `kind:0`, `content` = `{name: slug, …}` + `["host", "<hostname>"]` | Replaceable, stored. NMP keeps the latest — this is the canonical `pubkey → slug + host` source. |
| `Presence` (heartbeat) | `kind:30315` + `["h", project]` + `["d", "tenex-edge-presence:<session-id>"]` + `["p", whitelisted]…` + `["agent", pubkey, slug]` + `["session-id", id]` + `["expiration", …]` | Stored replaceable heartbeat, session-scoped by `d`; expires if unrefreshed so stale relay replays do not resurrect dead sessions. |
| `Activity` (distilled narrative) | `kind:1`, `content` + `["h", project]` | Stored note in the NIP-29 project group → durable activity log. |
| `Status` (live "currently…") | `kind:30315` (NIP-38), `["h", project]`, `["d", project]`, content (empty = idle), `["expiration", …]` (NIP-40) | Addressable/replaceable → only latest per project; expires if unrefreshed (crash safety). |
| `Mention` | `kind:1`, `content` + `["h", project]` + `["p", destPubkey]` + `["agent", senderPubkey, senderSlug]` + optional `["session-id", targetSession]` | Stored mention in the NIP-29 project group. Authorship implicit (signed by sender). Session tag = route to one session. |

**Every event type — heartbeat included — is codec-owned.** Don't assume any of them is
transport-fixed: another relay policy or Marmot shape would express presence/status/mention differently, so the
codec set must be swappable as a whole, per-event.

Litmus test for an honest seam: the domain layer names no concrete Nostr kind, tag, or wire-protocol concept.

## 4. Config, identity, project resolution

**`~/.tenex/config.json`** (device-level):
```json
{ "relays": ["wss://relay.tenex.chat"], "whitelistedPubkeys": ["<hex>", "<hex>"] }
```

**Agent keystore** — `--agent <slug>` → keypair, persisted per slug under `~/.tenex/agents/<slug>.json`,
created on first use, stable forever after.

> **Identity grain — decided: identity = (agent, machine).** An agent runs on **one machine**. The
> same tool on two machines (claude-code on `backend1` vs `backend2`) is **two different agents with
> two pubkeys** — ideally two names the user chose. So host-on-`kind:0` is correct and coherent: the
> profile *is* per-(agent, machine), and `host` lets a peer see which machine it's talking to. This
> does **not** violate "value attaches to the identity, not the host" — the **session** is the
> interchangeable body; the agent identity is durable across all sessions on its machine. Resolves
> open **Q8** for M1: durable per-agent (machine-bound), ephemeral per-session.

**`.tenex/project.json`** (optional, per-repo): `{ "slug": "tenex-edge" }`

**Project slug resolution order:**
1. `.tenex/project.json` `slug`, searched cwd → git root.
2. else git repo name (toplevel basename) — so **all worktrees share one slug**.
3. else basename of `$PWD`.

## 5. Session start & the heartbeat

`tenex-edge session-start --agent <slug>` (run by the SessionStart hook, which passes cwd + CC's
`session_id` on stdin):

1. Resolve config, agent key, project slug; adopt CC's `session_id`.
2. **Ensure the agent's `kind:0` is published** (name = slug, `host` tag = hostname) so peers can
   resolve it. Idempotent — only republish on change.
3. Write the session record to SQLite (binds `session_id → agent slug/pubkey, project, CC PID`).
4. **Return fast on stdout** (never block the editor), then fork the detached per-session process.

The forked process publishes the **presence heartbeat every 30s** (the session-scoped `kind:30315` shape in §3),
refreshes status, and runs the subscriber/reaper (§7).

## 6. Commands (host-neutral CLI surface)

| Command | Does |
|---|---|
| `session-start --agent <slug>` | §5: publish `kind:0`, record session, fork the per-session process, print session id. |
| `session-end --session <id>` | Clean stop: kill the process, publish empty status. Graceful path. |
| `send-message --recipient <target> "<msg>" --session <my-id>` | Emit a `Mention`. **Sender** = the agent bound to `--session`. **`--recipient`** is either a **session id (or unique prefix)** → routed to that one session, or an **agent name** (`slug` / `slug@project`) → routed to that agent (all its active sessions; queued if none — see §8). |
| `who [project-slug] [--live]` | List peers (from `kind:0` + live presence). For peers with **active work**, show a **session-id prefix** so the agent can target a specific session. `--live` opens a continuously refreshing terminal board over the same local awareness snapshot. |
| `inbox --session <id>` | Drain + print pending mentions for a session (used by the injection hook, §8). |
| `tail -f [project-slug]` | Subscribe via the codec, decode every event type, print colorized: profiles, presence join/leave, activity, status, mentions. Read-only; exercises the full decode path. |

## 7. Runtime — per-session process, resolution, liveness, routing

**One detached process per session**, each embedding its own **NMP kernel** (single actor thread +
relay connections + NIP-42 AUTH + LMDB store). It is **bidirectional**:

- **Publishes:** presence (30s), status, auto-distilled activity.
- **Subscribes:** the NIP-29 project group's open traffic (`h` = project); feeds the local directory
  and writes mentions addressed to this agent into the SQLite inbox.

**Resolution (simpler now that `kind:0` exists):**
- `agent name → pubkey + host`: from `kind:0` via NMP's store — **stored and queryable**, never
  stale. (No longer dependent on catching a live presence heartbeat.)
- `who is live / which sessions`: from session-scoped **presence** heartbeats; expiry prevents stale
  stored events from counting as live.
- `--recipient <session-id>`: resolved against live presence to the owning pubkey + session.

**Liveness / reaper** (SessionEnd is graceful-only — not guaranteed on crash):
- Capture CC's PID **before** detaching (`getppid()` is useless post-daemonize — reparented to init).
  Poll `kill(pid,0)` each tick; if gone → publish empty status, exit.
- Presence and status both carry **NIP-40 expiration**. On detected death, the reaper publishes an
  already-expired presence heartbeat plus empty status — belt and suspenders.

**Mention routing / multi-session injection** (the core of your session-targeting ask):
- Mention with a **`session-id` tag** → only that session's process injects it.
- Mention with **no session tag** (agent-level) → delivered to **all active sessions** of that agent
  in that project.
- Agent-level mention with **no active session** → **queued** in SQLite, delivered when that agent
  next starts a session. *(Auto-launching a Claude Code in the right directory is future, not M1.)*
- *Build note:* two sessions of the same agent = two processes on the same pubkey, both subscribed to
  the same `p`-tag. Make the inbox **idempotent, keyed by `(mention_event_id, target_session)`**, so
  an agent-level mention received by both processes injects once per session, never twice.

> **NMP embedding — verify (§10):** N concurrent per-session processes each want an NMP kernel.
> Confirm how NMP prefers to be embedded many-times-per-device — separate LMDB store paths per
> process vs a shared/daemon mode — since LMDB is single-writer per env. This is the one place the
> "per-session process" choice meets NMP's assumptions.

## 8. Awareness, mentions, injection

**Activity — auto-distilled.** Agents forget to narrate themselves, so we don't ask them to. Host
adapters mark turn start/end and provide the live transcript path; tenex-edge reads the recent
conversation transcript and **distills** it into one-line intents ("fixing auth", "opened a PR"),
publishing `Activity` + refreshing `Status`. Distillation lives *inside* tenex-edge (domain logic);
the hook only flips turn state and supplies transcript context.

> **Distillation — decided: LLM-based, exactly like `pc`.** Intent-level lines aren't recoverable from
> tool calls heuristically, so the distiller is an **LLM step** over conversation context. The codec
> and publish path are unchanged; only the distiller is LLM-backed.

**Status (NIP-38)** — `kind:30315`, `["h", project]`, `["d", project]`, content = current distilled
activity, empty when idle; refreshed by the process; NIP-40 expiration.

**Mentions** — `send-message` resolves *me* from `--session` (the skill injects `${CLAUDE_SESSION_ID}`),
resolves the recipient per §6/§7, and publishes the `Mention`.

**Injection (in scope).** A **UserPromptSubmit hook** runs `tenex-edge inbox --session ${CLAUDE_SESSION_ID}`
and injects any pending mentions as context before the agent's turn. The same hook also injects a
compact **peer-awareness block** (from `who`) — including the **session-id prefixes** for peers with
active work — so the agent knows who's around and how to target a specific session. The actual
injection is the host's job (context-injection role); tenex-edge only exposes `inbox`/`who`.

## 9. The skill

Project skill at `.claude/skills/tenex-send-message/SKILL.md`, frontmatter `description` +
`allowed-tools: Bash`. Body tells the agent it can mention another agent or a specific session, with
the session id substituted in:

```
tenex-edge send-message --recipient <session-id | agentSlug[@project]> "<message>" --session ${CLAUDE_SESSION_ID}
```

`${CLAUDE_SESSION_ID}` is a documented skill substitution — how the short-lived command recovers
"who am I" despite the session id not being in the Bash environment. Activity needs no skill (it's
auto-distilled); discovery is handled by the injected peer-awareness block (§8).

## 10. Risks / must-verify

1. **NMP embedding for N per-session processes** — store/connection model under concurrency (§7).
   The highest-uncertainty integration point; settle it before committing to per-session processes.
2. **NIP-42 AUTH** — largely handled by `nmp-nip42`; confirm `relay.tenex.chat`'s requirement and
   that NMP authenticates with our agent key as expected.
3. **Presence expiry behavior** — verify relays return `kind:30315` presence heartbeats with
   `expiration` intact and that clients ignore expired replays before marking sessions live.
4. **SessionEnd not guaranteed on crash** — designed around via the reaper (§7).
5. **Transcript availability** — confirm each host adapter can provide a live transcript snapshot
   while a turn is running, so distillation is based on conversation context.

## 11. Decisions — resolved

> **Build update (2026-06-08): NMP deviation, validated.** On inspection NMP is a
> full cross-platform *app kernel* (Elm-architecture single-actor thread, FFI,
> flatbuffers codegen, reactive UI projections) — built for SwiftUI/Compose/wasm
> apps, a poor fit for a headless fire-and-forget CLI daemon. Embedding it would
> dominate the build. Since the wire output is identical standard Nostr and
> transport sits behind the codec seam, M1 was built on the lean `nostr-sdk`
> stack instead; an NMP-backed transport (and its Marmot/MLS crates
> as future codec shapes) remains a drop-in. Result: working binary, 50 unit
> tests + a real-relay e2e test, and live demos (multi-agent + a real `claude -p`
> session publishing presence/activity). See `README.md`.

| # | Decision |
|---|---|
| Language | **Rust**, on `nostr-sdk` (NMP deferred behind the codec seam — see note above). |
| Activity | **Auto-distilled** from the conversation transcript via turn-start/turn-end hooks (not agent-narrated). |
| Injection | **In scope** for M1 (UserPromptSubmit hook drains the session inbox). |
| Keystore | Per-slug key files under `~/.tenex/agents/`, generated on first use. |
| Local state | **SQLite** for app state (sessions, inbox, queue, PID); NMP's LMDB for protocol events. |
| Process model | **One process per session.** |
| Naming | Domain event is **`Mention`** (not "direct message"). |
| Identity grain | **identity = (agent, machine)** — an agent runs on one machine; the same tool on another machine is a different agent/pubkey (user-named). host-on-`kind:0` confirmed. Resolves Q8 for M1. |
| Distillation | **LLM-based, like `pc`** — triggered by turn state and fed by transcript context. |

**Open — small:** exact `host` tag value (hostname vs stable device id); whether agent-level mentions
fan out to *all* active sessions or a primary (recommend all, §7).

## 12. Suggested build order

1. `config` + `identity` (keystore) + `project` resolution — no network, testable immediately.
2. `domain` models/events + the `kind1` `codec` set — pure, unit-testable, no relay.
3. `transport`: embed NMP; prove publish/subscribe + NIP-42 AUTH against the relay with a throwaway
   event; confirm the per-process embedding model (§10.1).
4. `tail` — first end-to-end decode path; eyes for everything after.
5. `session-start`: publish `kind:0`, record session, fork the per-session process (presence + status
   + reaper).
6. Subscriber → `kind:0`/presence resolution → `who` → `send-message` (`Mention`) with session-level
   targeting.
7. PostToolUse → auto-distilled activity/status.
8. Inbox + UserPromptSubmit injection (mentions + peer-awareness block); the skill.
